Project Overview

A full-stack churn prediction system for a streaming service. Users are classified into one of three churn categories using an XGBoost multiclass classifier trained in Python on historical data pulled from PostgreSQL. The trained model is deployed to a FastAPI backend on Render, which queries a separate set of test tables — stored in the same PostgreSQL database — to serve live predictions to the dashboard.

Goal

Predict which subscribers are likely to leave — split into three outcomes (Renews Cancels Early Won't Renew) — enabling targeted retention strategies rather than treating all at-risk users the same.

Database Design

The project uses a single PostgreSQL database with two sets of tables — training tables and test tables — all connected via user_id as a foreign key. The schema separates concerns clearly: user demographics, subscription info, engagement behaviour, support history, and billing are each stored independently.

Seeded by seed_database.py. Includes a churn_labels table with the ground truth used to train the model.

Connection Setup

The database connection is handled in database.py using psycopg2. Credentials are never hardcoded — they are loaded from a .env file using python-dotenv, keeping sensitive information out of the codebase and out of version control.

from dotenv import load_dotenv import os, psycopg2 load_dotenv() def get_connection(): connection = psycopg2.connect( host = os.getenv("DB_HOST"), database = os.getenv("DB_NAME"), user = os.getenv("DB_USER"), password = os.getenv("DB_PASSWORD"), port = os.getenv("DB_PORT") ) return connection

The .env file is never committed to version control (added to .gitignore). For EDA notebooks, the connection was upgraded to SQLAlchemy to avoid a pandas deprecation warning with raw psycopg2 connections — the same credentials are reused.

Database Seeding

Both databases were created entirely in Python — no SQL client or GUI was used. This makes setup fully reproducible: clone the project, configure .env, and run the script to recreate the database from scratch.

How seed_database.py works
  • Generates user data (age, country, signup_date) for all n users and collects them in a list first.
  • Inserts all users in a single batch using execute_values — much faster than inserting one row at a time.
  • Retrieves all generated user_ids via RETURNING user_id, then builds related records for all other tables.
  • Inserts each table's data in one bulk call.
  • Wraps everything in a single transaction — if seeding fails halfway through, a rollback ensures no partial data is left.
Execution flow
def main(): connection = get_connection() cursor = connection.cursor() try: create_tables(cursor) # CREATE TABLE IF NOT EXISTS for all tables seed_users(cursor, n=10000) # bulk insert users + all related tables connection.commit() except Exception as e: connection.rollback() # clean slate on any failure raise finally: cursor.close() connection.close()

Churn Label Logic

Churn is assigned probabilistically using a weighted score to avoid purely deterministic labels. A fully deterministic rule would allow the model to memorise the logic rather than learn from the features.

# Weighted churn score with noise churn_score = ( (days_since_last_watch > 40) * 0.5 + # strongest signal (tickets > 3) * 0.3 + # support signal (price_increase and watch_hours < 5) * 0.2 # billing signal ) churn_prob = churn_score + random.uniform(-0.05, 0.05) # add noise # Multiclass label assignment if not churn: churn_type = 0 # Renews elif cancelled: churn_type = 1 # Cancels Early (cancelled before renewal) else: churn_type = 2 # Won't Renew (lets subscription lapse)

1Cancels EarlyCancels before the current period ends.
2Won't RenewLets the subscription lapse at expiry without cancelling early.

Feature Engineering

Feature engineering transforms raw data into more informative inputs. After this step the dataset grew from 15 to 31 columns. The same build_features() logic runs in both the training notebook and in main.py at serving time, ensuring full consistency.

engagement_score
df['engagement_score'] = ( df['completion_rate'] * 0.4 + (df['avg_watch_hours_per_week'] / 20) * 0.4 + (df['number_of_logins_per_week'] / 20) * 0.2 )

A weighted composite of watch hours (40%), completion rate (40%), and logins (20%). Gives the model one clean pre-combined signal instead of three correlated features.

tenure_days
df['tenure_days'] = (reference_date - df['signup_date']).dt.days

How long the user has been subscribed, in days. Captures loyalty — long-standing users tend to be stickier. Requires signup_date to be converted to datetime first (DATE columns from PostgreSQL are read as object dtype by pandas and must be converted manually).

is_inactive
df['is_inactive'] = (df['days_since_last_watch'] > 30).astype(int)

Binary flag (0/1) for users inactive for over 30 days. Directly encodes the strongest EDA signal — days_since_last_watch had the highest churn correlation at r = +0.37. Converting to a binary threshold makes the decision boundary explicit. 49% of users are flagged as inactive.

high_risk_billing
df['high_risk_billing'] = ( (df['price_increase_last_6m'] == True) & (df['payment_failures'] > 0) ).astype(int)

Flags users who have experienced both a price increase and a payment failure — a compound billing risk signal.

high_support
df['high_support'] = (df['tickets_last_30d'] > 3).astype(int)

Binary flag for users with more than 3 support tickets in the last 30 days, indicating friction or dissatisfaction.

is_yearly
df['is_yearly'] = (df['billing_cycle'] == 'yearly').astype(int)

Binary encoding of billing cycle. Annual subscribers are generally stickier, making this relationship explicit for the model.

effective_monthly_price
df['effective_monthly_price'] = np.where( df['billing_cycle'] == 'yearly', df['yearly_price'] / 12, df['monthly_price'] )

Normalises price to a monthly equivalent regardless of billing cycle, making price comparable across all users.

days_until_renewal
df['days_until_renewal'] = ( df['next_renewal_date'] - reference_date ).dt.days.clip(lower=0)

How many days until the user's next renewal. Users close to renewal with low engagement are at higher churn risk.

One-Hot Encoding
df = pd.get_dummies(df, columns=['country', 'plan'], drop_first=False)

country and plan are one-hot encoded. drop_first=False keeps all categories. The exact column order is saved to feature_columns.json after training, so the API aligns features identically at prediction time via align_columns().

Model Details

Three classifiers were trained and compared — Logistic Regression, Random Forest, and XGBoost — all evaluated on a held-out test set of 2,000 users (80/20 stratified split). XGBoost was selected as the final model based on its best macro F1 score. After training, the pipeline and exact feature column order are persisted to disk.

Key Takeaways
  • Renews and Won't Renew are predicted reliably — F1 of 0.92 and 0.69 respectively. These are the actionable majority classes for retention campaigns.
  • Cancels Early (F1 = 0.07) is the hardest class by far — only 75 examples in the test set (3.7%). Even with sample weighting, the signal is too sparse for confident predictions. This is a known limitation of the synthetic data design.
  • Random Forest achieved the highest ROC-AUC (0.9178) but scored 0.000 F1 on Cancels Early, meaning it never predicted that class at all. XGBoost's balanced approach makes it the better choice for real-world use.
  • days_since_last_watch is the single most important feature by a wide margin — consistent with the EDA correlation analysis.

JS Functions

loadFilters()async

Fetches available filter values from GET /filters and populates the three dropdown selects (country, plan, billing cycle) with <option> elements.

loadPredictions()async

Reads current filter values, builds a query string, and fetches GET /predict?…. On success, calls updateSummary(), updateBarChart(), and renderTable(). Manages loading state visibility.

side effectsSets allPredictions, updates summary cards, bar chart, and table.
updateSummary(s)

Reads the summary object returned by the API and triggers animated count transitions on all four summary card values.

s.totalTotal user count across the current filter set.
s.renews / s.cancels_early / s.wont_renewPer-class user counts.
animateCount(el, target)

Smoothly transitions a numeric text value from its current number to target over 500 ms using cubic ease-out via requestAnimationFrame.

elDOM element whose textContent is the animated counter.
targetFinal integer value to animate toward.
updateBarChart(s)

Rebuilds the bar chart HTML inside #bar-chart from the summary object. Each bar's width is set as an inline percentage, triggering the CSS transition on .bar-fill.

renderTable(predictions)

Clears and rebuilds the predictions table using a DocumentFragment for performance. Each row gets a colored .tag badge based on predicted_churn_type (0, 1, or 2).

predictionsArray of prediction objects from the API.
lookupUser()async

Reads the user ID from #lookup-id and calls GET /predict/:id. Renders a result card with country, plan, billing cycle, predicted class badge, and all three class probabilities as pill badges.

showLoading(show)

Toggles visibility of the #table-loading overlay and clears the table body when entering loading state.

showboolean — true to show loading, false to hide.
